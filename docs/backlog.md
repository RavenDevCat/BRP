# Backlog

Moved from the live project to keep deferred ideas visible in the unified structure.

- Reframe the platform around current-scheme assessment and optimization advice, not only route generation.
  - Detailed implementation direction:
    - see `docs/route-reallocation-design.md`
  - Core workflow direction:
    - import the supplier's or operator's current school-bus scheme into the system
    - evaluate whether the current design is operationally reasonable
    - produce concrete optimization suggestions rather than only replacing the whole plan with a new one
  - Immediate business goal:
    - assess whether supplier fare increases are justified
    - check whether current route design, bus count, line count, and vehicle mix are reasonable
    - determine whether the same rider population truly requires the currently proposed number of vehicles and routes
  - Assessment outputs to add over time:
    - current-state versus system-benchmark comparison
    - route efficiency review
    - vehicle mix rationality review
    - stop count, route count, fleet count, total distance, and total time comparison
    - route-to-route reallocation recommendations
      - identify when a stop or a small group of stops would fit better in a neighboring route
      - explain the reason in operational language, such as lower average route time, lower average route distance, better load balance, or the ability to remove one route entirely
      - present the recommendation as an actionable suggestion instead of only a score, for example: move stops A and B from Route 7 to Route 5 to reduce total operating effort and avoid one underutilized vehicle
    - theoretical minimum fleet / route baseline under fixed service constraints
    - conservative, moderate, and aggressive optimization recommendations
  - Negotiation support potential:
    - produce an internal benchmark independent of the supplier's proposal
    - estimate whether supplier pricing is aligned with actual operational complexity
    - support discussions with data rather than manual intuition
  - Product growth potential:
    - expand from route design into an operations audit, optimization-advice, and decision-support platform
    - add pricing analysis, service-level analysis, and scenario comparison modules in later phases
- Add traffic time multiplier for time-of-day adjustments.
  - Current design note:
    - see `docs/traffic-assumptions-design.md`
  - Version 1 direction:
    - profile-based travel-time multipliers only
    - do not modify distance
    - start with `Off-Peak`, `AM Peak`, and `PM Peak`
    - add city-aware defaults where the workbook clearly belongs to a supported city
  - Future refinement direction:
    - city-specific traffic profiles
    - school-direction profiles such as `AM To School` and `PM From School`
    - calibration from historical operating data
- Add large-vehicle road access constraints.
- Evaluate multi-profile OSRM by vehicle type.
- Improve concurrent cache behavior.
  - Current state:
    - geocode and subway caches are shared across users/tasks
    - cache writes are now atomic, so JSON corruption risk is reduced
    - but concurrent cache misses for the same new address can still trigger duplicate API requests
  - Future improvement options:
    - add single-flight / in-progress key deduplication for geocode and subway lookups
    - or introduce per-job temporary caches with a controlled merge-back strategy
- Add lightweight access control and user directory for small-scale internal rollout.
  - Preferred architecture direction:
    - use Cloudflare Access as the primary login / gatekeeping layer for `client.example.com`
    - keep an application-side user directory for identity mapping, roles, and audit history
    - avoid building a custom password system unless scale or requirements change
  - Suggested app-side data model:
    - `users`: email, display name, role, status, created_at, last_login_at
    - `planner_runs`: user_id, source_file_name, created_at, status, elapsed_seconds
    - `audit_logs`: user_id, action, created_at, detail
  - Storage direction:
    - start with SQLite
    - do not place the SQLite file in the OneDrive-synced project tree
  - Security tightening to add later:
    - do not leave the backend publicly exposed without an auth gate
    - prefer front-end access through Cloudflare Access and a restricted backend trust path
